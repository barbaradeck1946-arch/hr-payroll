Introduction

Zeri HR is a complete HR management system built for managing employee records, attendance, leave, payroll, notices, approvals, teams, projects and role-based access control.

The system is designed for companies that need a structured HR workflow with permission-based visibility. Each user sees only the modules and records allowed by their assigned role permissions.

Installation

Local Setup

  1. Place the project folder inside the web root, for example htdocs/hr-payroll.
  2. Create a MySQL database.
  3. Copy .env.example to .env and configure database credentials.
  4. Install dependencies with Composer.
  5. Run migrations and the default admin seeder.
  6. Open the application in the browser.
composer install
php artisan key:generate
php artisan migrate
php artisan db:seed --class=AdminUserSeeder

Important Storage Commands

php artisan storage:link
php artisan optimize:clear
After changing Blade files or configuration, clear cache if the browser does not show the latest updates.

Employee Login Setup

Creating an employee profile does not automatically create login access. Login access comes from the user account, and the employee profile must be linked to that user account.

Required Workflow

  1. Go to User Management > Add User.
  2. Create a user with name, email, password and role.
  3. Use the Employee role for normal employee access, or assign another role based on your permission setup.
  4. Approve or activate the user account if approval is enabled.
  5. Go to Employees > Employee List.
  6. Edit the employee profile.
  7. In the User Account field, select the user account created earlier.
  8. Save the employee profile.
  9. The employee can now log in using the linked user email and password.

How It Works Internally

employees.user_idLinks the employee profile to the login user account.
If an employee has no linked user account, the employee cannot log in even if the employee profile exists.

Payroll Overview

The Payroll module manages salary structure, employee-specific salary assignment, bonuses, loans, deductions, provident fund and payslip generation.

Professional payroll flow: create salary grade, create reusable salary template, assign employee-specific salary snapshot, generate payroll draft, HR reviews, final submit, then mark payments as paid.

Payroll Flow

  1. Create Salary Grades.
  2. Create reusable Salary Templates.
  3. Assign Salary Template to Employee and customize individual salary values.
  4. Add bonuses, loans, deductions and provident fund settings.
  5. Generate Payroll Draft for selected period. Draft shows expected salary, bonus, loan installment deduction, other deductions, PF, tax and net payable.
  6. Review all payroll items and payslips before final submission.
  7. Final Submit payroll run. This is the posting step for salary-linked records such as loan installment payment and PF contribution.
  8. Mark individual payslips as paid after payment processing.

Payroll Run Generation Logic

StepSystem Logic
Period checkThe system checks pay frequency, period start and period end. If a draft run exists for the same period, it is regenerated. If a processed run exists, only a missing selected employee can be added. Approved or paid runs cannot be regenerated.
Employee selectionIf no employee is selected, payroll is generated for all active employees. If an employee is selected, payroll is generated only for that active employee.
Salary sourcePayroll first uses the employee-specific salary assignment active at the payroll period end date. If no assignment exists, it falls back to the employee salary grade minimum salary.
AllowancesHouse rent, medical allowance, conveyance allowance and other allowance are taken from the employee salary assignment.
BonusBonus amount is included when the bonus date falls inside the payroll period.
Loan deductionOnly active loans are considered. Pending installments with due dates inside the payroll period are summed into the draft payslip.
Other deductionsActive employee deduction records are attached to the payslip when their effective date range and frequency match the payroll period.
PF and taxProvident fund and tax are calculated from configured percentages, usually from the employee salary assignment or PF setup.
Draft totalsGross, deduction total and net payable are calculated for each payslip and summarized in the payroll run.
Loan balances are not reduced during draft generation. Loan installments are marked paid only when the payroll run is finalized, because the deduction becomes real only when salary is processed.

Employee Loans

The Loan module tracks employee loan requests, approval status, installment schedules and salary-based repayment.

Loan Master

  • Employee.
  • Loan reference.
  • Principal amount.
  • Interest percentage.
  • Installment count.
  • Installment amount.
  • Issued date and first installment date.
  • Status: pending supervisor, pending final, active, paused, closed or rejected.

Loan Approval Flow

  1. Employee submits a loan request with loan.apply or employee_loan.apply.
  2. Supervisor or department approver reviews team requests with loan.approve-supervisor or employee_loan.approve-supervisor.
  3. Final approver completes approval with loan.approve-final or employee_loan.approve-final.
  4. After final approval, the loan becomes active and installment schedules are generated.
  5. Rejected loans are stored with rejection information and are not deducted from payroll.

Loan Installments

Installments are created from the approved loan schedule. Payroll picks pending installments when their due date falls inside the payroll period.

Salary Deduction Workflow

  1. Loan must be active before it can be deducted from payroll.
  2. Payroll draft calculates loan_deduction from pending installments due inside the payroll period.
  3. HR reviews the draft payslip and confirms that the loan deduction is correct.
  4. When HR final submits the payroll run, those due installments are marked as paid.
  5. The paid installment stores the payroll item reference so HR can see which salary/payslip paid it.
  6. When all installments are paid, the loan status becomes closed automatically.
Recommended workflow: employee applies for loan, supervisor approves, final approver activates it, payroll deducts due installments from salary, and final payroll updates loan balances.
If loan installments are manually paid, rescheduled or changed after payroll draft generation, regenerate the payroll draft before final submission so payslip deduction and loan balance remain the same.

Manual Installment Payment

Manual installment payment should be used only for payments collected outside salary, such as cash, bank deposit or adjustment. Salary-related repayment should normally happen through payroll finalization.

Payroll Runs

Payroll run is the monthly or weekly payroll batch for one period.

Draft and Final Submission Flow

  1. HR generates payroll draft for a period.
  2. The system creates payroll items for employees.
  3. HR reviews every payroll item and payslip.
  4. If needed, HR adjusts employee salary assignment, bonus, loan, deduction or PF setup.
  5. HR regenerates draft for the same period.
  6. When everything is correct, HR clicks Final Submit.
  7. The run becomes processed, PF transactions are posted and due loan installments are marked paid against their payslips.

Payroll Calculation

Gross = basic_salary + allowance_total + bonus_total
Total Deduction = loan_deduction + other_deduction + provident_fund_deduction + tax_deduction
Net Payable = Gross - Total Deduction

Final Submit Posting Rules

  • Payroll draft can be regenerated before final submission.
  • Final submission locks the payroll run as processed.
  • Due pending loan installments in the payroll period are marked paid and linked to the employee payslip.
  • If due loan installment amounts changed after draft generation, the system blocks final submission and asks HR to regenerate the draft.
  • Provident fund contribution transactions are posted for the run.
  • After final submission, payslips can be marked as paid when salary payment is completed.
Finalized payroll runs cannot be regenerated for the same period. Create a new period or review the finalized run.

Users, Roles and Permissions

User Management controls who can access the system and what each user can do.

Users

  • Create user accounts.
  • Approve or reject user access.
  • Assign users to roles.
  • Link users with employee records.

Roles

  • The default seed creates common organization roles with practical permission assignments.
  • Super Admin and Admin receive all permissions by default.
  • HR Admin, HR Manager, Payroll Manager, Finance Manager, Department Head, Supervisor, Project Manager, Team Lead, Auditor and Employee receive scoped default permissions.
  • Admin or HR users can change the assigned permissions later from the Roles screen.
  • Users inherit access from assigned roles.

Permissions

Permissions are grouped by module, for example employee.view, payroll_run.generate, payslip.view and settings.update.

Permission access labels are database driven. Each permission stores an access scope key, label, badge class and description, so the role permission screen can show whether a permission is for own access, department/team access, admin/global access or general access.

Data Access Rules

  • Employee self-service: Employees can view their own attendance, leave applications, leave balances, assigned teams, assigned projects, assigned tasks, payslips, loans, deductions and provident fund records when the related view permission is assigned.
  • Supervisor access: Supervisors can view their own records and direct subordinate records for attendance, leave approval/reporting and task work scope. Supervisor access does not automatically expose payroll finance records.
  • Payroll privacy: Salary, payslip, loan, deduction and provident fund records are self-only unless payroll management or payroll report permissions are assigned.
  • Work module access: Team, project and task lists are scoped to records where the user is a lead, manager, member, assignee, creator or direct supervisor of an assignee. Management permissions allow wider setup access.
  • Holiday access: Users with holiday view permission can see the holiday list/calendar. Create, update and delete actions require separate holiday permissions.
ScopeMeaning
Own / SelfUser can access their own records or personal actions.
Department / TeamUser can access assigned team, department or approval-scope records.
Admin / GlobalUser can access company-wide records, setup, approval, payroll or reports.
GeneralBasic module visibility or shared access.

Localization

Zeri HR supports multilingual UI labels for dashboard, employees, departments, designations, attendance, leave, payroll, users, roles, permissions, holidays, notices, teams, projects, settings and reports.

Supported Languages

🇫🇷 FrenchfrCore HR, payroll and admin UI labels.
🇮🇳 HindihiCore HR, payroll and admin UI labels.

How Language Selection Works

  • Users can change the interface language from the topbar language selector.
  • The selected language is stored in the session and applied through the locale middleware.
  • Translation files are stored in lang/en.json, lang/de.json, lang/fr.json and lang/hi.json.
  • Database content such as employee names, department names, project names and notice titles remains user-entered content and is not automatically translated.
When adding new screens, wrap visible static labels with Laravel translation helpers and add the keys to all supported language JSON files.

FAQ

Can employees in the same salary grade have different salaries?

Yes. Salary Grade defines the allowed range. Employee Salary Assignment stores the actual employee-specific salary snapshot.

Does payroll use the shared salary template directly?

No. Payroll uses the employee-specific salary snapshot stored in employee_salary_templates.

Can HR review payroll before final submission?

Yes. Payroll generation creates a draft run. HR reviews the run and final submits only after checking.

Can payment be marked before payroll final submission?

No. Payslip payment can be marked only after the payroll run is finalized.

Why can some users not see a module?

The user role may not have the required permission. Assign the needed permission from Role Permissions.
